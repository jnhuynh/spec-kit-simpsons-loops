# Data Model: Phase-Aware Specs with Splitting Skill

**Date**: 2026-05-28
**Feature**: `c31c-feat-phase-aware-specs`

## Entities

### Phase Annotation

**Location**: `## Phases` section within parent spec.md
**Created by**: `/speckit.specify` (phase-aware generation)
**Consumed by**: Splitting skill (`/speckit.split`)

| Property | Type | Description |
|----------|------|-------------|
| Phase Number | Integer (1-10) | Ordered position in the deployment sequence |
| Slug | Kebab-case string | Short descriptive label for the phase (e.g., `expand-schema`, `payment-integration`) |
| Assigned Stories | List of references | Story identifiers from the User Scenarios section assigned to this phase |
| Release Strategy | Enum | `dark launch with gradual reveal` or `direct release` |
| Boundary Rationale | Free text | Explanation of why this phase boundary exists |

**Markdown format within spec.md**:

```markdown
## Phases

### Phase 1: expand-schema
**Stories**: User Story 1, User Story 2
**Release Strategy**: dark launch with gradual reveal
**Rationale**: Database schema changes require expand-and-contract migration sequencing. Must be deployed and validated before dependent features can build on the new schema.

### Phase 2: payment-integration
**Stories**: User Story 3
**Release Strategy**: dark launch with gradual reveal
**Rationale**: Third-party payment provider integration needs production validation with limited traffic before full rollout.

### Phase 3: ui-reveal
**Stories**: User Story 4, User Story 5
**Release Strategy**: direct release
**Rationale**: User-facing UI changes that depend on the schema and integration being stable. Low risk additive work.
```

**Validation rules**:
- Phase numbers must be sequential starting from 1 with no gaps
- Each user story must be assigned to exactly one phase
- Every phase must have a release strategy and rationale
- Maximum 10 phases per parent spec (FR-012a)
- Phase slug must be valid kebab-case (lowercase alphanumeric and hyphens)

### Manifest

**Location**: `## Manifest` section at the end of parent spec.md
**Created by**: Splitting skill (`/speckit.split`)
**Updated by**: Splitting skill (on re-run), developer (manual status updates)

| Property | Type | Description |
|----------|------|-------------|
| Phase | String | Phase number and slug (e.g., `P1: expand-schema`) |
| Directory | String | Child spec directory name relative to `specs/` |
| Description | Free text | Brief description of the phase scope |
| Status | Enum | `Draft`, `In Progress`, `Complete`, or `Cancelled` |
| Release Strategy | String | `dark launch with gradual reveal` or `direct release` |

**Markdown format within spec.md**:

```markdown
## Manifest

| Phase | Directory | Description | Status | Release Strategy |
|-------|-----------|-------------|--------|------------------|
| P1: expand-schema | c31c-feat-billing-overhaul--p1-expand-schema | Database schema expansion for new billing fields | Draft | dark launch with gradual reveal |
| P2: payment-integration | c31c-feat-billing-overhaul--p2-payment-integration | Stripe payment provider integration | Draft | dark launch with gradual reveal |
| P3: ui-reveal | c31c-feat-billing-overhaul--p3-ui-reveal | Billing UI components and user flows | Draft | direct release |
```

### Status State Machine

**Valid values**: Draft, In Progress, Complete, Cancelled

**Transitions**:

```
[Draft] --start--> [In Progress]
[In Progress] --finish--> [Complete]
[Draft] --cancel--> [Cancelled]
[In Progress] --cancel--> [Cancelled]
```

**Invalid transitions** (rejected by splitting skill on re-run):
- In Progress -> Draft (no rollback)
- Complete -> Draft (no rollback)
- Complete -> In Progress (no reopening)
- Cancelled -> Draft (no resurrection)
- Cancelled -> In Progress (no resurrection)
- Cancelled -> Complete (no resurrection)

### Parent Spec

**Location**: `specs/{ticket-type-description}/spec.md`
**Type**: Standard SpecKit spec with two additional sections (Phases, Manifest)

| Property | Type | Description |
|----------|------|-------------|
| Standard spec sections | Markdown | User Scenarios, Requirements, Success Criteria, etc. (unchanged) |
| Phases section | List of Phase Annotations | Phase groupings generated by the specify step |
| Manifest section | Table | Child spec tracking generated by the splitting skill |

**States**:

```
[Standard Spec] --/speckit.specify with multi-concern description--> [Phase-Annotated Spec]
[Phase-Annotated Spec] --/speckit.split--> [Phase-Annotated Spec with Manifest]
[Phase-Annotated Spec with Manifest] --developer edits phases, re-runs /speckit.split--> [Updated Manifest]
```

**Validation rules**:
- A spec with Phases but no Manifest has been annotated but not yet split
- A spec with a Manifest but no Phases is invalid (manifest depends on phase annotations)
- Pipeline steps (`/speckit.plan`, `/speckit.tasks`, etc.) do not run on parent specs that have been split; they operate on individual child specs

### Child Spec

**Location**: `specs/{parent-directory-name}--p{N}-{phase-slug}/spec.md`
**Type**: Standard SpecKit spec (identical structure to any standalone spec)
**Created by**: Splitting skill (`/speckit.split`)

| Property | Type | Description |
|----------|------|-------------|
| spec.md | Markdown file | Standard SpecKit spec populated with the stories, requirements, and success criteria relevant to this phase |
| Directory name | String | `{parent-directory-name}--p{N}-{phase-slug}` |
| Phase number | Integer | Extracted from directory name (the N in `p{N}`) |
| Parent reference | String | Extracted from directory name (everything before `--`) |

**Naming convention**:

```
Parent:  c31c-feat-billing-overhaul
Child 1: c31c-feat-billing-overhaul--p1-expand-schema
Child 2: c31c-feat-billing-overhaul--p2-payment-integration
Child 3: c31c-feat-billing-overhaul--p3-ui-reveal
```

**Validation rules**:
- Directory name must match pattern `{parent}--p{N}-{slug}` where N is 1-10
- Total directory name must not exceed 200 characters (slug truncated if needed)
- `spec.md` must pass the existing spec quality checklist as a standalone spec
- Child spec is fully independent -- can run through the complete SpecKit pipeline without referencing siblings

**States**:

```
[Not Exists] --/speckit.split--> [Created with Draft status]
[Created] --developer edits--> [Modified]
[Modified] --/speckit.split re-run--> [Reconciled] (may include conflict markers)
[Any active state] --phase removed from parent--> [Preserved on disk, Cancelled in manifest]
```

### Conflict Marker

**Location**: Inline within child spec.md content
**Created by**: Splitting skill during reconciliation (re-run)
**Resolved by**: Developer (manual editing)

| Property | Type | Description |
|----------|------|-------------|
| Opening marker | HTML comment | `<!-- CONFLICT: {description of triggering change} -->` |
| Closing marker | HTML comment | `<!-- END CONFLICT -->` |
| Original content | Markdown | The child spec content that conflicts with the propagated change, preserved between markers |

**Example**:

```markdown
<!-- CONFLICT: P1 child spec changed the authentication method from OAuth2 to API keys. This section assumed OAuth2 token management. -->
### Authentication Flow

Users authenticate via OAuth2 token exchange with the payment provider...

<!-- END CONFLICT -->
```

## Relationships

```
speckit.specify ──generates──> spec.md (with Phases section for multi-concern features)
speckit.specify ──references──> spec-template.md (with new Phases section guidance)

speckit.split ──reads──> parent spec.md (Phases section)
speckit.split ──creates/updates──> child spec directories (specs/{parent}--p{N}-{slug}/)
speckit.split ──creates/updates──> child spec.md (within each child directory)
speckit.split ──writes──> parent spec.md (Manifest section)

parent spec.md ──contains──> Phase Annotations (Phases section)
parent spec.md ──contains──> Manifest (tracking table)
parent spec.md ──references──> child spec directories (via Manifest)

child spec.md ──may contain──> Conflict Markers (after reconciliation)
child spec.md ──feeds into──> speckit.plan, speckit.tasks, speckit.homer, speckit.lisa, speckit.ralph, speckit.marge

Manifest ──tracks──> child spec status (Draft, In Progress, Complete, Cancelled)
Phase Annotations ──assign──> user stories to phases (one story per phase)
```
