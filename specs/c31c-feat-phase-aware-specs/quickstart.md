# Quickstart: Phase-Aware Specs with Splitting Skill

## Implementation Overview

This feature touches 3 existing files and creates 2 new source files. Changes are organized into two workstreams:

1. **Phase-Aware Specify** -- Modify the specify command and spec template to generate phase annotations
2. **Splitting Skill** -- New command and agent for splitting phase-annotated specs into independent child specs

| File | Change | Priority |

## Implementation Sequence

### Phase 1: Phase-Aware Specify (P1 -- User Story 1)

**Step 1**: Create source copy of `speckit.specify.md` in `speckit-commands/` from the installed copy at `.claude/commands/speckit.specify.md`. This follows the project convention where source files live in `speckit-commands/` and `setup.sh` copies them to `.claude/commands/`.

**Step 2**: Update `speckit-commands/speckit.specify.md` to add phase-aware generation logic:
- After step 4 (Fill User Scenarios) and before step 5 (Generate Functional Requirements), add a new step that analyzes user stories for deployment boundaries
- Add phase detection heuristics: migration terms, integration terms, infrastructure patterns, story count/complexity
- Add phase grouping logic: assign each story to exactly one phase, determine release strategy per phase
- Add phase section generation: write `## Phases` section with structured metadata
- For single-concern features, generate a single phase with `direct release`

**Step 3**: Update `.specify/templates/spec-template.md` to include a `## Phases` section placeholder between User Scenarios and Requirements:

```markdown
## Phases

<!--
  This section is generated by /speckit.specify when the feature involves
  multiple deployment concerns. Each phase groups related user stories
  with a release strategy recommendation.
  
  For single-concern features, a single phase with "direct release" is generated.
  Remove this section entirely if not applicable.
-->
```

### Phase 2: Splitting Skill (P1 -- User Story 2)

**Step 4**: Create `speckit-commands/speckit.split.md` -- the splitting skill command file:
- Detect current feature directory (from branch name or user input)
- Read parent spec.md and parse the `## Phases` section
- Validate: phases exist, count <= 10, slugs are valid
- For each phase, create child spec directory under `specs/` with naming convention `{parent}--p{N}-{slug}`
- Generate child spec.md for each phase with standard SpecKit structure
- Create or update `## Manifest` section in parent spec
- Handle idempotent re-runs: detect existing children, update rather than recreate

**Step 5**: Create `claude-agents/split.md` -- the splitting skill agent file:
- Single-shot agent (like plan.md, tasks.md)
- Invokes `/speckit.split` command
- Commits and pushes results

**Step 6**: Update `setup.sh` to copy the new command and agent files:
- Copy `speckit-commands/speckit.split.md` to `.claude/commands/speckit.split.md`
- Copy `claude-agents/split.md` to `.claude/agents/split.md`

### Phase 3: Chain Awareness and Reconciliation (P2 -- User Story 3)

**Step 7**: Extend `speckit-commands/speckit.split.md` with reconciliation logic:
- On re-run, read current state of all earlier child specs
- Compare earlier child spec content against what was originally generated
- For later child specs, identify sections affected by earlier-phase changes
- If later child spec section has no manual edits, update it in place
- If later child spec section has manual edits that conflict, insert conflict markers
- Report conflicts to developer

### Phase 4: Status Tracking (P2 -- User Story 4)

**Step 8**: The manifest table already includes a Status column (from Step 4). This step adds:
- Status transition validation in the splitting skill (reject invalid transitions on re-run)
- Clear documentation in the command file about valid status values and transitions

### Phase 5: Phase Management (P3 -- User Story 5)

**Step 9**: Extend `speckit-commands/speckit.split.md` with phase management:
- Detect added phases (in parent annotations but not in manifest) -- create new child directories
- Detect removed phases (in manifest but not in parent annotations) -- mark as Cancelled, preserve directory
- Maintain correct phase ordering in manifest after adds/removes

## Phase-Aware Specify Logic (Pseudocode)

```
function analyze_phases(user_stories):
    signals = detect_signals(user_stories)
    
    if no migration_signals AND no integration_signals AND no infrastructure_signals:
        return single_phase(all_stories, "direct release")
    
    groups = []
    
    # Group 1: Foundation (migrations, schema changes)
    foundation_stories = stories with migration/schema signals
    if foundation_stories:
        groups.append(Phase(foundation_stories, "dark launch with gradual reveal"))
    
    # Group 2: Integration (third-party, external services)
    integration_stories = stories with integration signals
    if integration_stories:
        groups.append(Phase(integration_stories, "dark launch with gradual reveal"))
    
    # Group 3: Dependent features (build on foundation/integration)
    dependent_stories = stories depending on groups 1 or 2
    if dependent_stories:
        groups.append(Phase(dependent_stories, assess_risk(dependent_stories)))
    
    # Group 4: User-facing reveal (UI, public endpoints)
    reveal_stories = remaining stories
    if reveal_stories:
        groups.append(Phase(reveal_stories, "direct release"))
    
    return number_phases(groups)
```

## Splitting Skill Logic (Pseudocode)

```
function split(parent_spec_path):
    spec = read(parent_spec_path)
    phases = parse_phases_section(spec)
    
    validate(phases.count <= 10)
    validate(phases.numbers_sequential)
    validate(all_stories_assigned_once(phases, spec.stories))
    
    parent_dir_name = basename(dirname(parent_spec_path))
    existing_manifest = parse_manifest(spec)  # may be empty
    existing_children = find_child_directories(parent_dir_name)
    
    for phase in phases:
        child_dir = f"{parent_dir_name}--p{phase.number}-{phase.slug}"
        child_path = f"specs/{child_dir}/spec.md"
        
        if child_dir in existing_children:
            reconcile(child_path, phase, earlier_phases)
        else:
            create_child_spec(child_path, phase, spec)
    
    # Handle removed phases
    for existing in existing_children:
        if existing not in phases:
            set_status(manifest, existing, "Cancelled")
    
    update_manifest(parent_spec_path, phases, statuses)
```
