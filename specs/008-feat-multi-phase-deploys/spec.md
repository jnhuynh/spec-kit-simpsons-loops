# Feature Specification: Multi-Phase Deploy Support

**Feature Branch**: `008-feat-multi-phase-deploys`
**Created**: 2026-04-27
**Status**: Draft
**Input**: User description: "Add multi-phase deploy support to the speckit + Simpsons-loops pipeline so a single feature can ship as a stack of independently-deployable PRs."

## Clarifications

### Session 2026-04-27

- Q: When the split step fails partway through opening a stack of pull requests (for example, `gh pr create` succeeds for phase 1 but errors on phase 2 because of a network or API failure), what should the split step do? → A: Fail fast on first error, leave successfully-created branches and pull requests in place, and rely on idempotent re-runs to complete the stack.
- Q: How does the split step discover review findings so it can enforce the per-phase gating rule in FR-018, given that today's review step prints findings to stdout only and does not persist them? → A: The review step persists its report to `<FEATURE_DIR>/review-report.md` on every run; the split step reads that file to identify unresolved high-severity findings and their phase tags. Absence of the file means the split step refuses to run and instructs the author to run review first.
- Q: What concrete values are valid for the severity, phase tag, and resolution status of each finding in the persisted review report (FR-012a), so that the split step can deterministically interpret it? → A: severity is one of `high|medium|low|informational`; phase tag is the integer phase number when multi-phase and omitted when single-phase; resolution status is one of `open|resolved`, with new findings emitted as `open` and only marked `resolved` by a subsequent review pass against the same branch state. The split step treats anything other than `resolved` as gating.
- Q: Does the split step's per-phase gating rule in FR-018 apply to every high-severity finding regardless of which check pack produced it, or only to high-severity migration-safety findings (as SC-006 currently implies)? → A: The split step gates on every unresolved high-severity finding regardless of which check pack produced it. SC-006 is updated to match FR-018; the persisted review report does not need a check-pack discriminator field for gating. Only the severity and resolution status determine whether a finding gates the split step.
- Q: How should the split step group a commit on a multi-phase feature branch that has no `Phase:` trailer (for example, an infrastructure commit inserted by the implementation agent that has no corresponding tagged task), given the apparent conflict between FR-014 ("default to phase 1") and an earlier assumption that suggested inheriting the phase of the most recent prior phase-tagged commit? → A: Untrailerd commits MUST always be grouped into phase 1 by the split step, deterministically and without consulting neighboring commits. FR-014 is authoritative; the conflicting "inherit prior phase" clause is removed from the Assumptions section. Authors who need an infrastructure commit to land in a phase other than phase 1 MUST add the appropriate `Phase: N` trailer manually (or rebase to attach it).
- Q: How does the split step "update existing phase branches in place" (FR-017, US4 AS2/AS3) when the cherry-picked commit set changes — given that re-cherry-picking produces new commit SHAs and any change to history requires a force-push that would also overwrite human edits if a phase branch were edited manually? → A: For each phase, the split step deterministically rebuilds the phase branch by resetting it to the appropriate base (`main` for phase 1, the previous phase branch for phase K>1) and re-cherry-picking that phase's current commits from the feature branch, then force-pushes the rebuilt phase branch to the remote with `--force-with-lease`. Phase branches are pipeline-managed artifacts: humans MUST NOT commit directly to `NNNN-<type>-<slug>-phaseK` branches, and the split step's force-push will overwrite any such edits. The split step identifies existing phase branches and pull requests by the deterministic naming convention `NNNN-<type>-<slug>-phaseK` (FR-015) — a remote branch matching that name is treated as an existing phase branch to update; no remote branch with that name means a new phase branch to create. If a phase branch's pull request is already merged into `main`, the split step MUST skip rebuilding that phase (treat it as immutable history) and report the skip; this is the only safeguard against the spec-freeze edge case where a tasks-artifact `[phase-N]` tag is changed after commits with that tag have already shipped.
- Q: For single-phase features, findings in the persisted review report carry no phase tag (per FR-012a), so FR-018's "unresolved high-severity finding tagged to it" wording leaves single-phase gating undefined. Does the split step gate single-phase features on unresolved high-severity findings, and if so, how? → A: The split step MUST gate the single pull request for a single-phase feature on every unresolved high-severity finding in `<FEATURE_DIR>/review-report.md`, regardless of phase-tag presence. For single-phase features, "any phase" in FR-018 and SC-006 means the implicit single phase that contains the entire feature; a finding without a phase tag is treated as belonging to that implicit phase. The split step's behavior on unresolved high-severity findings is identical for single-phase and multi-phase features: refuse to open or update the affected pull request and report which finding blocked it. This makes safety gating uniform across both modes and prevents the regression where a single-phase feature could ship with unresolved high-severity findings.
- Q: How does the split step report its outcome (success, gating refusal, idempotent skip, partial failure) so authors and the pipeline orchestrator can determine what happened without re-running it? → A: The split step MUST persist a structured outcome report to `<FEATURE_DIR>/split-report.md` on every run (overwriting any prior copy) and MUST mirror a concise human-readable summary to stdout. The persisted report MUST list every phase enumerated by the run with its terminal status — one of `created` (new phase branch and pull request opened), `updated` (existing phase branch rebuilt and pull request updated in place), `unchanged` (existing phase branch and pull request matched the recomputed state and no force-push was required), `skipped-merged` (phase branch's pull request was already merged into `main` per FR-017), `gated` (phase blocked by an unresolved high-severity finding per FR-018), or `failed` (the split step failed fast on this phase per FR-017's fail-fast rule) — together with the resulting branch name, pull-request URL when applicable, and a single-sentence reason for any non-success status. For single-phase features the report contains exactly one entry for the implicit single phase. The persisted report MUST be machine-readable enough that the pipeline orchestrator can extract `(phase number, status, reason)` for every entry without re-running the split step. Absence of the report after a split-step invocation MUST be treated by the pipeline orchestrator as a failure.
- Q: What concrete on-disk format must `<FEATURE_DIR>/review-report.md` and `<FEATURE_DIR>/split-report.md` use, given that FR-012a and FR-019a only require them to be "machine-readable enough" to extract specific fields? → A: Both files MUST use a single GitHub Flavored Markdown table as their machine-readable surface, with column headers fixed by the spec, one row per finding (review report) or per phase (split report), and pipe (`|`) characters in cell values escaped as `\|`. The review-report table columns MUST be exactly `| ID | Severity | Phase | Status | Check Pack | Summary |` in that order, where `ID` is a stable per-finding identifier the review step assigns and reuses across runs, `Phase` is the integer phase number or the literal `-` for single-phase features, `Status` is `open` or `resolved`, and `Check Pack` is the source check pack filename (informational; not used for gating per the earlier clarification). The split-report table columns MUST be exactly `| Phase | Status | Branch | PR URL | Reason |` in that order, where `Phase` is the integer phase number or the literal `single` for single-phase features, `PR URL` is the pull-request URL or `-` when omitted, and `Reason` is the single-sentence explanation or `-` when omitted. Both files MAY contain additional human-readable prose before and after the table, but consumers MUST locate the table by its exact header row and parse rows until the next blank line or end of file. This makes the persisted reports parseable with standard Unix tooling (`awk -F '|'`) without introducing a new dependency.
- Q: What severity must the migration-safety check pack assign to each of the unsafe patterns enumerated in FR-010, given that the split step's gating rule (FR-018, SC-006) only triggers on `high` severity and the spec does not currently pin a severity to any pattern? → A: Every unsafe pattern enumerated in FR-010 (NOT NULL column adds without default; column drops while prior-phase code still reads them; single-phase renames not following the four-step expand-contract pattern; long-transaction backfills on hot tables; missing index for a new read path; phases containing both a schema change and code that depends on the new schema; removed function or endpoint while prior-phase callers still exist; per-phase deployability violations) MUST be emitted by the migration-safety check pack as `high` severity, because each pattern describes a production-breaking condition for zero-downtime deployments. This makes the gating contract uniform: any FR-010 pattern detected on the integrated branch will gate the affected phase per FR-018. The migration-safety check pack MAY emit lower-severity (`medium`, `low`, `informational`) findings for advisory observations that fall outside the FR-010 catalog (e.g., stylistic guidance on migration scripts), but FR-010-cataloged patterns MUST always be emitted as `high`. SC-004 is updated to require that every FR-010 pattern, when present in a deliberately-broken plan, is reported as a `high`-severity finding tagged to the offending phase.
- Q: Which agent or check pack is responsible for emitting findings for the structural-consistency edge cases (orphan phase tag referencing an undeclared phase per Edge Case 1; non-contiguous phases such as commits tagged `Phase: 1` and `Phase: 3` with no `Phase: 2` per Edge Case 2; malformed `Phase:` trailer with a non-integer or empty value per Edge Case 3), so that the FR-018 gating mechanism acts on them via the persisted review report? → A: The review step (Marge) performs all structural-consistency checks against the integrated feature branch as part of the same run that loads the migration-safety check pack, and emits each detected violation as a `high`-severity finding into `<FEATURE_DIR>/review-report.md` via the migration-safety check pack (`migrations.md`), so the existing gating contract (FR-018) acts on them uniformly. The Phase column of each row MUST be set as follows: for an orphan phase tag, the integer phase number from the offending tag (e.g., `5` for `[phase-5]`); for a non-contiguous-phase gap, the integer of the missing phase (e.g., `2` when `Phase: 1` and `Phase: 3` exist with no `Phase: 2`); for a malformed `Phase:` trailer whose value cannot be parsed as a positive integer, the literal `-`. The split step MUST treat any `high`-severity finding whose Phase column is `-` as gating every phase in the run (a global gate), so a malformed trailer blocks the entire stack until the author fixes the trailer; this preserves uniform safety gating for unparseable inputs. The split step does NOT independently re-verify these structural invariants; it relies entirely on the persisted review report, consistent with the review-report-as-single-source-of-truth model established in earlier clarifications.
- Q: How should the pipeline behave when the plan artifact has no "Deploy Phases" section (single-phase per FR-002) but the feature branch contains one or more commits carrying a `Phase:` trailer — for example, an author removed the "Deploy Phases" section after implementation, or copied phase-trailered commits onto a single-phase branch? → A: This is a structural-consistency violation and MUST be detected by the review step (Marge) via the migration-safety check pack, emitted as a `high`-severity "phase-trailer-without-deploy-phases" finding into `<FEATURE_DIR>/review-report.md` with the Phase column set to the literal `-` so the split step treats it as a global gate per FR-018 and refuses to proceed for the entire feature until the inconsistency is resolved (either by adding a "Deploy Phases" section to the plan or by rewriting the offending commits to drop the `Phase:` trailer). The split step does NOT attempt to silently ignore trailers in single-phase mode or silently treat the feature as multi-phase: FR-002 remains authoritative on multi-phase detection (the plan's "Deploy Phases" section is the sole signal), and the review step is responsible for catching the inconsistency before split-step gating acts on it. This pattern is added to the FR-010 catalog so the gating contract remains uniform.
- Q: Which `main` ref does the split step use as its base for enumerating commits (FR-013), cherry-picking phase 1 (FR-015), and resetting phase 1 during idempotent rebuilds (FR-017) — the local `main` branch, the remote-tracking ref `origin/main`, or some other reference — given that local `main` may lag behind the remote and could yield non-deterministic stacks across runs from different working copies? → A: The split step MUST use the remote-tracking ref `origin/main` as the canonical base for every operation that the spec or assumptions describe as relative to "main": commit enumeration (`git log origin/main..HEAD` for FR-013), the phase 1 cherry-pick base (FR-015), the phase 1 reset target during idempotent rebuilds (FR-017), and the merge target referenced by `skipped-merged` detection (FR-017). To keep this base current, the split step MUST run `git fetch origin main` before enumerating commits or rebuilding any phase branch, so the working copy's `origin/main` ref reflects the remote tip used by reviewers and `gh pr create`. The split step MUST NOT consult the local `main` branch for any decision; this prevents non-determinism when the working copy's local `main` lags behind the remote and ensures two independent working copies produce identical stacks for the same feature branch and remote state. Throughout the spec, every reference to "main" (in FRs, assumptions, edge cases, and acceptance scenarios) means `origin/main` after this fetch unless explicitly qualified otherwise. If the `git fetch origin main` itself fails (e.g., network error, authentication failure), the split step MUST fail fast before any branch is rebuilt, emit a `failed` row in the split report with a reason that names the fetch failure, and instruct the author to retry once connectivity is restored — consistent with FR-017's fail-fast model.
- Q: What concrete title and body must the split step generate for each phase pull request opened or updated under FR-016, given that the spec does not currently specify either, and how does idempotent re-execution treat existing titles and bodies on phase pull requests that were edited by a human reviewer? → A: The split step MUST generate the title and body deterministically from the feature branch name and the plan artifact, and MUST overwrite both fields on every run so that title and body are pipeline-managed artifacts (parallel to FR-017's treatment of phase branches as pipeline-managed). For multi-phase features, the title MUST be exactly `[Phase K/N] <feature-branch-name>` where `K` is the phase number, `N` is the total phase count enumerated by the plan's "Deploy Phases" section, and `<feature-branch-name>` is the feature branch name (e.g., `008-feat-multi-phase-deploys`). For single-phase features, the title MUST be exactly `<feature-branch-name>`, matching today's behavior. The body MUST be a deterministic GitHub Flavored Markdown document composed of: (1) a first-line link of the form `Part of <FEATURE_DIR>/spec.md` using the feature directory's path relative to the repository root, so reviewers can locate the source spec; (2) for multi-phase features, a `## Phase Goal` section whose body is the goal text copied verbatim from the corresponding phase entry in the plan's "Deploy Phases" section; (3) for multi-phase features, a `## Post-deploy production state` section whose body is the post-deploy state text copied verbatim from the same phase entry; (4) for multi-phase features, a `## Stack` section that lists every phase in the stack as a bullet of the form `- Phase J: NNNN-<type>-<slug>-phaseJ` for J in 1..N, with the literal suffix ` (this PR)` appended to the bullet for `J == K`; (5) for single-phase features, only item 1 (matching today's minimal pull-request body). On every run, the split step MUST recompute the title and body, compare them against the current pull-request title and body via `gh pr view --json title,body`, and: report `unchanged` only when the recomputed title and body match the current pull-request fields AND the phase branch's commit SHAs match the recomputed phase branch SHAs (no force-push performed); otherwise overwrite the pull-request fields via `gh pr edit --title --body` and report `updated` (or `created` for a newly opened pull request). Phase pull-request titles and bodies are pipeline-managed: human edits to either field WILL be overwritten on the next split-step run, and reviewers requesting changes MUST use pull-request review comments rather than editing the description. This makes the `unchanged` status of FR-019a deterministically computable from `(commit SHAs, title, body)` and prevents drift between the plan artifact and the rendered pull-request stack.
- Q: How do the existing tasks-template "Stages" (the structure currently labeled "Phase 1: Setup", "Phase 2: Foundational", "Phase 3+: User Stories" per FR-005) coexist with the per-deploy-phase top-level sections that FR-004 mandates for multi-phase features — are Stages removed entirely, flattened into deploy phases, or nested as second-level headings inside each deploy phase? → A: For multi-phase features, deploy phases are the sole top-level (`##`) organizing structure of `tasks.md` per FR-004, and the existing Setup/Foundational/User Stories template structure is preserved as second-level (`###`) headings nested inside each deploy phase, relabeled as "Stages" per FR-005 to disambiguate them from deploy phases. Concretely, a multi-phase tasks.md uses the structure `## Phase 1: <goal>` followed by zero or more of `### Stage: Setup` / `### Stage: Foundational` / `### Stage: User Stories` (a stage heading MUST be emitted only when that stage actually contains tasks for that deploy phase; empty stages MUST be omitted rather than emitted as empty headings). Every task under any stage within deploy-phase K MUST carry the `[phase-K]` tag per FR-004. The tasks-generation agent MUST allocate Setup, Foundational, and User-Story work into the deploy phases that need it (a deploy phase MAY contain only User-Story tasks if it has no setup or foundational work of its own, and the same kind of work MAY appear in multiple deploy phases when each phase has its own setup or foundational needs). For single-phase features, the existing template is preserved unchanged: Setup/Foundational/User Stories remain at the top level (`##`), no "Stages" relabel is applied, and no `[phase-N]` tags are added. This makes the multi-phase tasks structure deterministic for the implementation agent (Ralph reads top-level headings to enumerate deploy phases and second-level headings to enumerate stages within a phase) and preserves the single-phase tasks template as a strict subset of today's behavior per FR-022.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Author a multi-phase feature end to end (Priority: P1)

A spec author needs to ship a change that cannot land in a single PR without breaking production — for example, renaming a database column read by a running service. The author writes one feature description that targets the desired end state. The pipeline plans the deploy phases needed to get there safely, implements every phase on a single feature branch, reviews the whole change with migration-safety checks, and finally splits the branch into a stack of per-phase pull requests that humans review and merge in order.

**Why this priority**: This is the primary capability the feature delivers. Without it, the only way to model a phased deploy today is by faking it as multiple unrelated specs, which fragments feature intent and provides no migration-safety guardrails. Every other story builds on this end-to-end flow.

**Independent Test**: Can be tested by running the full pipeline against a synthetic feature description that requires phased rollout (e.g., "rename `users.email_address` column to `users.email`"), and verifying that the pipeline produces (a) a single spec, (b) a plan that enumerates four deploy phases, (c) a tasks list where every task is tagged with its phase, (d) a feature branch whose commits each carry a `Phase: N` git trailer matching their task tag, and (e) a stack of four pull requests with the correct base-branch chain (`main` <- phase1 <- phase2 <- phase3 <- phase4).

**Acceptance Scenarios**:

1. **Given** a feature description that requires phased rollout, **When** the pipeline runs end to end, **Then** `plan.md` contains a "Deploy Phases" section enumerating phases 1..N with a goal and post-deploy production state for each phase.
2. **Given** a `plan.md` with a "Deploy Phases" section, **When** tasks are generated, **Then** every task in `tasks.md` carries a `[phase-N]` tag and tasks are organized under per-phase top-level sections.
3. **Given** a `tasks.md` with `[phase-N]` tags, **When** the implementation step commits each completed task, **Then** every commit message contains a `Phase: N` git trailer whose value matches the task's phase tag.
4. **Given** a fully-implemented feature branch with phase-tagged commits and a passing review, **When** the split step runs, **Then** stacked branches `NNNN-feat-name-phase1`, `NNNN-feat-name-phase2`, ... `NNNN-feat-name-phaseK` are created and a stack of pull requests is opened with the correct base-branch chain.

### User Story 2 - Catch unsafe phase boundaries before splitting (Priority: P1)

When a multi-phase feature is reviewed, the reviewer agent must apply migration-safety checks that look at each phase as a deployable slice on top of the previous one. A plan that drops a column in the same phase that adds it, that adds a NOT NULL column without a default, that contains both a schema change and the code that depends on it, or that renames a column in a single phase instead of the safe four-step expand-contract pattern must be flagged as a finding tagged with the phase that introduced the problem. The split step refuses to open a PR for a phase that has unresolved high-severity findings against it.

**Why this priority**: The whole point of multi-phase deploys is migration safety. Without per-phase migration-safety enforcement, the feature is just bookkeeping that splits commits by tag — it provides no guarantee that the resulting PR stack is actually safe to merge in order. Catching unsafe boundaries before the split step prevents the pipeline from publishing a known-broken stack.

**Independent Test**: Can be tested by deliberately authoring a broken plan (drop column in the same phase that adds the new one; no dual-write step) and confirming that the review step flags the expected migration-safety findings (rename in single phase; column dropped while prior-phase code still reads it; phase contains both schema change and dependent code) with high severity, that each finding is tagged with the offending phase, and that the split step refuses to open a PR for the broken phase until findings are resolved.

**Acceptance Scenarios**:

1. **Given** a plan that drops a column in the same phase that adds the replacement, **When** the review step runs, **Then** a high-severity finding is emitted, the finding is tagged with the offending phase, and the finding cites the migration-safety check it failed.
2. **Given** an unresolved high-severity finding tagged to phase K, **When** the split step runs, **Then** the split step refuses to open a pull request for phase K and reports which finding blocked it.
3. **Given** a plan that adds a NOT NULL column without a default, **When** the review step runs, **Then** the finding identifies the affected phase and references the relevant migration-safety check.
4. **Given** a phase that contains both a schema change and code that depends on the new schema, **When** the review step runs, **Then** the review emits a finding directing the author to split the schema change into an earlier phase and the code switch into a later phase.

### User Story 3 - Single-phase features keep working unchanged (Priority: P1)

Authors of features that do not need phased rollout (the vast majority) must see no change in behavior. A `plan.md` without a "Deploy Phases" section continues to produce a single feature branch and a single pull request, with no phase trailers required on commits, no per-phase findings emitted by the reviewer, and no extra PRs opened by the split step.

**Why this priority**: Backward compatibility is non-negotiable. Every existing feature in `specs/` was authored under the single-phase assumption. Breaking any of them — by requiring phase tags, by changing commit-message conventions, by inserting an extra split step that opens duplicate PRs — would invalidate the project's existing pipeline and force a migration of completed work. Backward compat must be structural (driven by the absence of a "Deploy Phases" section), not gated behind a flag.

**Independent Test**: Can be tested by re-running the pipeline against an existing single-phase feature (e.g., `001-consistency-cleanup`) and confirming that the output is the same shape as before: one branch, one PR, no `Phase:` trailers in commits, no per-phase findings in the review output, and no additional branches or PRs created by the split step.

**Acceptance Scenarios**:

1. **Given** a `plan.md` without a "Deploy Phases" section, **When** the pipeline runs, **Then** the implementation step commits without `Phase:` trailers and the review step does not emit per-phase finding tags.
2. **Given** a feature branch with no `Phase:` trailer on any commit, **When** the split step runs, **Then** exactly one pull request is opened against `main` (matching today's behavior) and no additional stacked branches are created.
3. **Given** a `tasks.md` without `[phase-N]` tags, **When** the implementation step processes a task, **Then** the resulting commit omits the `Phase:` trailer rather than defaulting silently to `Phase: 1` in the commit text.

### User Story 4 - Re-run the split step idempotently (Priority: P2)

When the split step runs more than once on the same feature branch — because the author re-ran the pipeline after fixing review findings, or because a phase needed an additional commit — re-running the split must update existing stacked branches and existing pull requests in place rather than creating duplicates. The author must be able to iterate on a multi-phase feature without manually pruning stale branches or stale PRs.

**Why this priority**: Pipelines re-run all the time. If split is not idempotent, every re-run leaves orphaned branches and orphaned PRs behind, and the human reviewer cannot tell which stack is current. Idempotence is what makes the split step safe to invoke automatically from the pipeline without requiring a "first run vs. re-run" distinction.

**Independent Test**: Can be tested by running the pipeline to completion on a multi-phase feature, observing the resulting stack of branches and PRs, then adding one new phase-tagged commit to the feature branch and re-running the split step. The expected result is that the corresponding phase branch is updated to include the new commit, the corresponding pull request is updated in place, and no additional branches or PRs are created.

**Acceptance Scenarios**:

1. **Given** a feature whose split step has already produced a stack of phase branches and PRs, **When** the split step runs again with no new commits, **Then** the existing branches and PRs are unchanged and no duplicates are created.
2. **Given** a feature whose split step has already produced a stack, **When** a new phase-tagged commit is added to the feature branch and the split step runs again, **Then** the corresponding phase branch is updated to include the new commit and the corresponding PR is updated in place.
3. **Given** a re-run that would shrink a phase (a commit was removed from the feature branch), **When** the split step runs, **Then** the corresponding phase branch is updated to reflect the new commit set without leaving orphaned commits.

### Edge Cases

- A phase tag references a phase number not declared in `plan.md` (e.g., a task tagged `[phase-5]` when the plan only enumerates phases 1..4). The review step (Marge) MUST emit this as a `high`-severity orphan-phase-tag finding via the migration-safety check pack per FR-010, with the Phase column set to the integer phase number from the offending tag. The split step then refuses to open a PR for that phase per FR-018 until the finding is resolved.
- The feature branch contains commits with `Phase: 1`, `Phase: 3`, and no `Phase: 2` (a phase has no commits). The review step MUST emit this as a `high`-severity non-contiguous-phases finding via the migration-safety check pack per FR-010, with the Phase column set to the integer of the missing phase (e.g., `2`). The split step then refuses to proceed for that phase per FR-018 rather than silently producing a non-contiguous stack.
- A single commit body contains a `Phase:` trailer with a non-integer or empty value. The review step MUST emit this as a `high`-severity malformed-trailer finding via the migration-safety check pack per FR-010, with the Phase column set to the literal `-`. The split step treats this as gating every phase per FR-018 and refuses to proceed until the trailer is fixed.
- The plan artifact has no "Deploy Phases" section (single-phase per FR-002) but the feature branch contains one or more commits carrying a `Phase:` trailer. The review step MUST emit this as a `high`-severity phase-trailer-without-deploy-phases finding via the migration-safety check pack per FR-010, with the Phase column set to the literal `-`. The split step treats this as gating every phase per FR-018 and refuses to proceed until the inconsistency is resolved (either by adding a "Deploy Phases" section to the plan or by rewriting the offending commits to drop the `Phase:` trailer).
- A task's `[phase-N]` tag in `tasks.md` is changed after some commits with that tag have already shipped. Spec freeze applies once Phase 1 has merged and deployed; this rule is documented but not enforced at the spec/plan/tasks level. The split step's only structural safeguard is FR-017: when a phase branch's pull request is already merged into `main`, the split step skips rebuilding that phase and reports the skip. The pipeline does not otherwise prevent re-tagging unshipped commits.
- The author runs the pipeline against a feature that produces only one phase even though a "Deploy Phases" section is present. The split step must produce exactly one PR (matching single-phase behavior) rather than an empty stack.
- A phase contains code that references symbols that will only be introduced in a later phase. The review step must flag this as a per-phase deployability violation.
- Cherry-picking a phase's commits onto the previous phase branch produces a merge conflict. The split step must report the conflict and stop rather than committing a partial stack; conflict resolution is a manual step in this iteration.
- Pull-request creation fails partway through the stack (for example, `gh pr create` succeeds for phase 1 but returns a network or API error on phase 2). The split step must fail fast on the first error, leave successfully-created phase branches and pull requests in place, report which phase failed and why, and rely on its idempotent re-run behavior (FR-017) to complete or repair the stack on the next invocation.

## Requirements *(mandatory)*

### Functional Requirements

#### Authoring and planning

- **FR-001**: The plan artifact MUST support an optional "Deploy Phases" section that enumerates phases 1..N, with a goal and a post-deploy production-state description for each phase.
- **FR-002**: The presence of a "Deploy Phases" section in the plan artifact MUST be the sole signal that marks a feature as multi-phase. Absence MUST mean single-phase.
- **FR-003**: When the plan identifies a multi-phase rollout, the planning agent MUST produce the "Deploy Phases" section in the plan artifact based on documented heuristics (schema migration combined with reads or writes on the same data, breaking API change, multi-service rollout coordination).
- **FR-004**: When the plan contains a "Deploy Phases" section, the tasks artifact MUST organize tasks under per-phase top-level (`##`) sections — one top-level section per deploy phase enumerated by the plan, in deploy order — and MUST tag every task with `[phase-N]` indicating which phase it belongs to. Deploy phases are the sole top-level organizing structure of multi-phase `tasks.md`; no other `##` sections are emitted.
- **FR-005**: When a feature is multi-phase, the existing per-story task structure (currently labeled "Phase 1: Setup", "Phase 2: Foundational", "Phase 3+: User Stories" at the top level of single-phase `tasks.md`) MUST be preserved as second-level (`###`) "Stage" headings nested inside each top-level deploy phase section, using the heading form `### Stage: Setup`, `### Stage: Foundational`, and `### Stage: User Stories`. The "Stage:" prefix disambiguates these subdivisions from deploy phases. A stage heading MUST be emitted only when that stage actually contains one or more tasks for the enclosing deploy phase; empty stages MUST be omitted rather than emitted as empty headings. The tasks-generation agent MUST allocate Setup, Foundational, and User-Story work into the deploy phases that need it (a deploy phase MAY contain only User-Story tasks if it has no setup or foundational work of its own, and the same kind of work MAY appear in multiple deploy phases when each phase has its own setup or foundational needs). For single-phase features, the existing labels and top-level placement MUST be preserved unchanged: Setup/Foundational/User Stories remain at the top level (`##`), no "Stages" relabel is applied, and no `[phase-N]` tags are added — preserving today's tasks template as a strict subset of multi-phase behavior per FR-022.

#### Implementation

- **FR-006**: When the implementation agent commits a task whose tasks-artifact entry carries a `[phase-N]` tag, the commit message MUST include a `Phase: N` git trailer whose value matches the task's phase tag.
- **FR-007**: When the implementation agent commits a task whose tasks-artifact entry does NOT carry a phase tag, the commit message MUST NOT include a `Phase:` trailer.
- **FR-008**: All implementation work for a multi-phase feature MUST occur on a single feature branch end to end before any pull request is opened.

#### Review

- **FR-009**: A migration-safety check pack MUST be available to the review agent and MUST be loaded automatically when present.
- **FR-010**: The migration-safety check pack MUST cover at minimum: NOT NULL column adds without default; column drops while prior-phase code still reads them; renames executed in a single phase rather than the four-step expand-contract pattern; long-transaction backfills on hot tables; missing index for a new read path; phases containing both a schema change and code that depends on the new schema; removed function or endpoint while prior-phase callers still exist; per-phase deployability (each phase must be deployable on top of `main + earlier phases` while previous-phase code is still running); orphan phase tags (a commit's `Phase: N` trailer or a task's `[phase-N]` tag references a phase number not declared in the plan's "Deploy Phases" section); non-contiguous phases (the set of phase numbers observed across commits or tasks contains a gap, e.g., `Phase: 1` and `Phase: 3` exist with no `Phase: 2`); malformed `Phase:` trailers (a commit body contains a `Phase:` trailer whose value cannot be parsed as a positive integer, including empty values); and phase-trailer-without-deploy-phases (the plan artifact has no "Deploy Phases" section per FR-002 but the feature branch contains one or more commits carrying a `Phase:` trailer, indicating an inconsistency between the multi-phase signal in the plan and the trailers attached during implementation; emitted with the Phase column set to the literal `-` because the inconsistency cannot be attributed to a specific phase, which causes the split step to treat it as a global gate per FR-018). Every finding emitted for one of these enumerated patterns MUST be assigned `high` severity, because each pattern describes a production-breaking condition for zero-downtime deployments or a structural-consistency violation that would yield a malformed PR stack. This makes the gating contract uniform: any FR-010 pattern detected on the integrated branch gates the affected phase per FR-018. The migration-safety check pack MAY emit lower-severity (`medium`, `low`, `informational`) findings for advisory observations that fall outside this catalog (e.g., stylistic guidance on migration scripts), but FR-010-cataloged patterns MUST always be emitted as `high`.
- **FR-011**: When the feature is multi-phase, every review finding MUST be tagged with the phase that introduced the issue.
- **FR-012**: The review step MUST run once on the whole integrated feature branch before the split step runs. Per-phase review MUST be expressed as findings tagged by phase, not as separate per-phase review passes.
- **FR-012a**: The review step MUST persist its findings report to `<FEATURE_DIR>/review-report.md` on every run, overwriting any prior copy. The machine-readable surface of the report MUST be a single GitHub Flavored Markdown table with the exact column headers `| ID | Severity | Phase | Status | Check Pack | Summary |` in that order, one row per finding, with pipe (`|`) characters in cell values escaped as `\|`. The file MAY contain additional human-readable prose before or after the table. Consumers MUST locate the table by its exact header row and parse rows until the next blank line or end of file. For each finding, the row MUST include:
  - **ID**: a stable per-finding identifier the review step assigns and reuses across runs against the same branch state, so a finding can be tracked from `open` to `resolved` without changing rows.
  - **Severity**: one of `high`, `medium`, `low`, `informational`.
  - **Phase**: the integer phase number that introduced the finding when the feature is multi-phase; the literal `-` when the feature is single-phase.
  - **Status**: one of `open` (the finding has not been addressed in the current branch state) or `resolved` (the finding was addressed in a subsequent review pass against the same branch state). New findings are emitted as `open`; the review step marks a previously-recorded finding as `resolved` only when a subsequent run confirms the underlying issue is gone.
  - **Check Pack**: the source check pack filename (informational; not used for gating per the earlier clarification that gating is determined by severity and resolution status alone).
  - **Summary**: a single-sentence human-readable description of the finding.

  The split step MUST treat any finding whose Status is anything other than `resolved` as gating per FR-018. The fixed table format makes the report parseable with standard Unix tooling (e.g., `awk -F '|'`) without introducing a new dependency.

#### Splitting

- **FR-013**: A new split step MUST read commits on the feature branch using `git log origin/main..HEAD` (after first running `git fetch origin main` to refresh the remote-tracking ref) and group them by their `Phase:` git trailer. The split step MUST NOT consult the local `main` branch; using `origin/main` as the canonical base ensures deterministic, identical output across independent working copies for the same feature branch and remote state.
- **FR-014**: Commits without a `Phase:` trailer MUST default to phase 1 for grouping purposes.
- **FR-015**: For each phase 1..N, the split step MUST create a stacked branch named `NNNN-<type>-<slug>-phaseK` (where `NNNN-<type>-<slug>` is the feature branch name and `K` is the phase number) by cherry-picking that phase's commits onto the previous phase's branch (or onto `origin/main` for phase 1).
- **FR-016**: The split step MUST open a pull request for every phase branch immediately, with the correct base-branch chain (`main` <- phase1 <- phase2 <- ... <- phaseN), so reviewers can read them in sequence. Each phase pull request's title and body MUST be deterministically generated by the split step and overwritten on every run (treating title and body as pipeline-managed artifacts, parallel to FR-017's treatment of phase branches). The title MUST be `[Phase K/N] <feature-branch-name>` for multi-phase features (where `K` is the phase number, `N` is the total phase count, and `<feature-branch-name>` is the feature branch name) and `<feature-branch-name>` for single-phase features. The body MUST be a deterministic GitHub Flavored Markdown document composed of: (1) a first-line link `Part of <FEATURE_DIR>/spec.md` using the feature directory's path relative to the repository root; (2) for multi-phase features, a `## Phase Goal` section copied verbatim from the goal text of the corresponding phase entry in the plan's "Deploy Phases" section; (3) for multi-phase features, a `## Post-deploy production state` section copied verbatim from the same phase entry's post-deploy state text; (4) for multi-phase features, a `## Stack` section listing every phase in the stack as `- Phase J: NNNN-<type>-<slug>-phaseJ` for J in 1..N, with the literal suffix ` (this PR)` appended for `J == K`; (5) for single-phase features, only item 1. The split step MUST recompute the title and body on every run, compare them against the current pull-request fields via `gh pr view --json title,body`, and overwrite them via `gh pr edit --title --body` when they differ. Human edits to a phase pull request's title or body WILL be overwritten on the next split-step run; reviewers MUST use pull-request review comments rather than editing the description. This makes title and body fully reproducible from the plan artifact and the feature branch name, and makes the `unchanged` status of FR-019a deterministically computable from the tuple `(commit SHAs, title, body)`.
- **FR-017**: The split step MUST be idempotent: re-running it MUST update existing phase branches and pull requests in place rather than creating duplicates. The split step identifies existing phase branches by the deterministic naming convention `NNNN-<type>-<slug>-phaseK` defined in FR-015; a remote branch matching that name is treated as an existing phase branch to update, and no remote branch with that name means a new phase branch to create. To update a phase branch in place when its commit set has changed (commits added, removed, or amended), the split step MUST deterministically rebuild the phase branch by resetting it to the appropriate base (`origin/main` for phase 1, the previous phase branch for phase K>1) and re-cherry-picking that phase's current commits from the feature branch, then force-push the rebuilt phase branch to the remote with `--force-with-lease`. Phase branches are pipeline-managed artifacts: humans MUST NOT commit directly to `NNNN-<type>-<slug>-phaseK` branches, and the split step's force-push will overwrite any such edits. If a phase branch's pull request is already merged into `origin/main`, the split step MUST skip rebuilding that phase (treat it as immutable history) and report the skip in its output. When the split step encounters an error while creating or updating a phase branch or its pull request (for example, a `gh pr create` failure on phase K), it MUST fail fast on the first error, leave already-completed phase branches and pull requests in place, and report which phase failed and why so that an idempotent re-run can resume from the failed phase. If the upstream `git fetch origin main` (required by FR-013 to refresh the canonical base) itself fails, the split step MUST fail fast before any phase branch is rebuilt, emit a `failed` row in the split report naming the fetch failure as the reason, and instruct the author to retry once connectivity is restored.
- **FR-018**: The split step MUST refuse to open or update a pull request for any phase that has an unresolved high-severity review finding tagged to it. The split step MUST read findings from the persisted review report at `<FEATURE_DIR>/review-report.md` (see FR-012a). If the review report is missing, the split step MUST refuse to run and instruct the author to run the review step first. The split step MUST report which finding blocked which phase. For single-phase features (where findings carry no phase tag per FR-012a), the split step MUST treat every unresolved high-severity finding as gating the single pull request and MUST refuse to open or update that pull request until the finding is resolved; this preserves uniform safety gating across single-phase and multi-phase modes. For multi-phase features, the split step MUST treat any unresolved high-severity finding whose Phase column in the persisted review report is the literal `-` (used by the migration-safety check pack to signal a structural violation that cannot be attributed to a specific phase, such as a malformed `Phase:` trailer per FR-010) as gating every phase in the run; the entire pull-request stack is blocked until the finding is resolved. For multi-phase features, when a finding gates phase K (whether by Phase = K or by global gate), the split step MUST also transitively gate every downstream phase K+1..N in the run, because each downstream phase's base branch is the previous phase's branch and cannot be sensibly built or updated when its base is itself gated; downstream phases MUST be reported as `gated` in the split report with a reason citing the upstream gating finding (e.g., "Downstream of gated phase K; resolve <finding-ID> then re-run."). The split step does NOT independently re-verify the structural invariants enumerated in FR-010; it relies entirely on the persisted review report as the single source of truth for gating decisions.
- **FR-019**: When the feature is single-phase (no "Deploy Phases" section in the plan), the split step MUST produce exactly one pull request against `main`, matching today's single-PR behavior, with no additional branches created. The single-phase split step remains subject to the gating rule in FR-018: if the persisted review report contains any unresolved high-severity finding, the split step MUST refuse to open or update the single pull request and report which finding blocked it.
- **FR-019a**: The split step MUST persist a structured outcome report to `<FEATURE_DIR>/split-report.md` on every run, overwriting any prior copy, and MUST mirror a concise human-readable summary to stdout. The machine-readable surface of the report MUST be a single GitHub Flavored Markdown table with the exact column headers `| Phase | Status | Branch | PR URL | Reason |` in that order, one row per phase enumerated by the run, with pipe (`|`) characters in cell values escaped as `\|`. The file MAY contain additional human-readable prose before or after the table. Consumers MUST locate the table by its exact header row and parse rows until the next blank line or end of file. For each phase enumerated by the run, the row MUST include:
  - **Phase**: the integer phase number when multi-phase; the literal value `single` when the feature is single-phase.
  - **Status**: one of `created` (new phase branch and pull request opened in this run), `updated` (existing phase branch rebuilt and/or pull-request title or body re-written in place), `unchanged` (the phase branch's commit SHAs already matched the recomputed phase branch SHAs AND the existing pull request's title and body already matched the recomputed values per FR-016, so no force-push and no `gh pr edit` were required), `skipped-merged` (the phase branch's pull request was already merged into `main` per FR-017), `gated` (the phase was blocked by an unresolved high-severity finding per FR-018), or `failed` (the split step failed fast on this phase per FR-017's fail-fast rule).
  - **Branch**: the phase branch name (`NNNN-<type>-<slug>-phaseK` when multi-phase; the feature branch name when single-phase), or the literal `-` only when the run failed before resolving the branch name.
  - **PR URL**: the URL of the corresponding pull request when one exists, or the literal `-` when omitted (for `gated` or `failed` entries that did not result in a pull request, and for `unchanged` entries when the existing pull-request URL was not re-fetched).
  - **Reason**: a single-sentence human-readable explanation, or the literal `-` when omitted. Required (non-`-`) for every non-success status (`skipped-merged`, `gated`, `failed`); optional for `created`, `updated`, and `unchanged`.

  The fixed table format makes the report parseable with standard Unix tooling (e.g., `awk -F '|'`) without introducing a new dependency. The pipeline orchestrator MUST treat absence of `<FEATURE_DIR>/split-report.md` after a split-step invocation as a split-step failure.

#### Pipeline orchestration

- **FR-020**: The pipeline orchestrator MUST detect whether a feature is multi-phase by checking the plan artifact for a "Deploy Phases" section. The orchestrator MUST NOT introduce a separate flag, environment variable, or command-line switch to indicate multi-phase mode.
- **FR-021**: The pipeline orchestrator MUST invoke the split step after the review step completes for both single-phase and multi-phase features. For single-phase features, the split step delegates to today's single-PR behavior.

#### Backward compatibility

- **FR-022**: Every existing single-phase feature MUST continue to work unchanged after this feature ships. Re-running the pipeline against an existing single-phase feature MUST produce the same shape of output (one branch, one PR, no phase trailers, no per-phase findings) as today.
- **FR-023**: The review agent's existing check-pack discovery MUST pick up the new migration-safety check pack automatically when present, without requiring code changes to the review agent.
- **FR-024**: The project's setup script (`setup.sh`) MUST seed the migration-safety check pack into downstream consumer projects idempotently — existing files MUST be preserved.

#### Documentation

- **FR-025**: The downstream-consumer template documentation MUST describe when to use multi-phase deploys, how a multi-phase feature is authored, and how to read the resulting pull request stack.
- **FR-026**: The plan artifact template MUST document the optional "Deploy Phases" section with a worked example.
- **FR-027**: The tasks artifact template MUST document phase tagging and the per-phase top-level structure used when a feature is multi-phase.
- **FR-028**: The project README MUST list multi-phase deploy support among the project's features.

### Key Entities

- **Deploy Phase**: A named step in a phased rollout. Has a phase number (1..N), a goal, and a description of the post-deploy production state. Lives as a section inside `plan.md`. Phases are totally ordered and contiguous.
- **Phase Tag**: A marker `[phase-N]` attached to a task in `tasks.md` indicating which deploy phase the task belongs to. References a phase declared in the plan's "Deploy Phases" section.
- **Phase Trailer**: A git trailer of the form `Phase: N` attached to a commit message indicating which deploy phase the commit belongs to. Set by the implementation agent based on the task's phase tag. Used by the split step to group commits.
- **Migration-Safety Check Pack**: A new check pack file (`migrations.md`) that the review agent loads to evaluate per-phase migration safety. Contains the catalog of unsafe patterns to detect and the rules for evaluating per-phase deployability.
- **Phase Branch**: A stacked branch produced by the split step, named `NNNN-<type>-<slug>-phaseK`. Contains the cherry-picked commits for one phase and is based on either `main` (phase 1) or the previous phase's branch.
- **Phase Pull Request**: A pull request produced by the split step for each phase branch. Forms a stack with the correct base-branch chain so reviewers read phases in deploy order.
- **Per-Phase Finding**: A review finding tagged with the phase number that introduced the issue. Used by the split step to gate which pull requests it is willing to open.
- **Review Report**: A persisted artifact at `<FEATURE_DIR>/review-report.md` produced by the review step on every run. Its machine-readable surface is a single GitHub Flavored Markdown table with the columns `| ID | Severity | Phase | Status | Check Pack | Summary |` (see FR-012a). Read by the split step to enforce per-phase gating without re-running review.
- **Split Report**: A persisted artifact at `<FEATURE_DIR>/split-report.md` produced by the split step on every run. Its machine-readable surface is a single GitHub Flavored Markdown table with the columns `| Phase | Status | Branch | PR URL | Reason |` (see FR-019a). Lists every phase enumerated by the run with its terminal status (`created`, `updated`, `unchanged`, `skipped-merged`, `gated`, `failed`). Read by the pipeline orchestrator to determine whether the split step succeeded without re-running it.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An author can take a feature description that requires phased rollout (such as a column rename) and produce a working stack of pull requests through the pipeline without manually editing branches, commit messages, or PR base-branch references.
- **SC-002**: For a multi-phase feature with K phases, the pipeline produces exactly K stacked pull requests with the correct base-branch chain on the first successful pipeline run, with zero manual cleanup steps required.
- **SC-003**: For every existing single-phase feature in the project's `specs/` directory, re-running the pipeline produces output of the same shape as today (one branch, one PR, no phase trailers in commits, no per-phase finding tags) — measured by spot-checking at least one existing feature end to end.
- **SC-004**: The migration-safety check pack catches every unsafe pattern enumerated in FR-010 when applied to a deliberately-broken plan that exhibits each pattern, with each finding emitted at `high` severity and tagged to the offending phase.
- **SC-005**: Re-running the split step against a feature that has already been split produces zero duplicate branches and zero duplicate pull requests; existing branches and pull requests are updated in place when their underlying commits change.
- **SC-006**: The split step refuses to open or update a pull request for any phase carrying an unresolved high-severity review finding (regardless of which check pack produced it), and the refusal message identifies which finding blocked which phase. For single-phase features, the split step refuses to open or update the single pull request whenever the persisted review report contains any unresolved high-severity finding.
- **SC-007**: A new author reading the project README and the downstream-consumer template can determine whether a feature they are about to write needs multi-phase deploys, and if so, how to author it, without consulting any other documentation.

## Assumptions

- The `gh` CLI is available in the pipeline environment and authenticated for the repository, since the split step opens pull requests via `gh pr create`. This matches the assumption made by the existing single-PR pipeline.
- The split step uses the remote-tracking ref `origin/main` (refreshed via `git fetch origin main` at the start of every run) as the canonical base for commit enumeration, phase 1 cherry-picking, phase 1 reset during idempotent rebuilds, and `skipped-merged` detection. The local `main` branch is never consulted, so two independent working copies of the same feature branch produce identical stacks for the same remote state. Throughout this spec, every reference to "main" means `origin/main` after this fetch unless explicitly qualified otherwise.
- All commits on a multi-phase feature branch are linear (no merges from `origin/main` mid-way through implementation), so cherry-picking by phase produces a clean stack. Resolving cherry-pick conflicts is treated as a manual step in this iteration; the split step reports the conflict and stops.
- Phases are totally ordered and contiguous (1, 2, 3, ... N with no gaps). A feature branch with commits tagged only `Phase: 1` and `Phase: 3` is a malformed input, not a supported configuration.
- The single global quality gate at `.specify/quality-gates.sh` applies to the integrated feature branch as a whole. There are no per-phase quality gates in this iteration.
- Spec freeze on first deploy is a documented social rule; the pipeline does not enforce it. Once Phase 1 of a feature has merged and deployed, further changes to the same feature's spec, plan, or tasks are out of scope for this iteration's tooling.
- The existing review agent's check-pack discovery loop picks up new check packs by filename, so dropping `migrations.md` into `.specify/marge/checks/` is sufficient to make it discoverable without modifying the agent's discovery logic.
- Review findings rated below high severity (medium, low, informational) do NOT block the split step from opening pull requests regardless of which check pack produced them; they are surfaced as review comments for the human reviewer to address. Only high-severity findings gate the split step, and they gate it regardless of which check pack produced them (migration-safety, baseline, project-specific, etc.).
- The project's existing branch-naming convention (`NNNN-<type>-<slug>`) is preserved for the feature branch. Phase branches extend this with a `-phaseK` suffix; this suffix lives within GitHub's 244-byte branch-name limit for all realistic phase counts.
- "One commit per task" remains the implementation agent's convention. Commits without a corresponding task entry (e.g., infrastructure commits inserted by the agent) MUST omit the `Phase:` trailer; the split step deterministically groups any untrailerd commit into phase 1 per FR-014. Authors who need such a commit to land in a different phase MUST attach the appropriate `Phase: N` trailer manually (or rebase to attach it) before invoking the split step.
- Cross-feature phase coordination, deploy automation, deploy-state tracking, and rollback automation are explicitly out of scope.
